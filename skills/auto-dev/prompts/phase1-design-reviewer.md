# 设计审查

## Task

审查设计文档 `{output_dir}/design.md`，按以下 checklist 逐项检查。

**Topic**: {topic}
**Language**: {language}

## Review Checklist

### 1. 目标对齐
- [ ] 问题陈述清晰（能一句话说清）
- [ ] 方案解决的是根因而非症状
- [ ] 范围合理（不过度设计，不遗漏关键点）
- [ ] 有成功标准（怎么判断设计达到目的）

### 2. 技术可行性
- [ ] 设计中引用的类/方法/接口在代码中真实存在（必须 grep 验证，不可凭记忆）
- [ ] API 签名正确（参数类型、顺序、返回值）
- [ ] 数据流可追踪（从输入到输出的完整路径）
- [ ] 依赖项可用（库、服务、配置）
- [ ] 无明显性能问题（N+1 查询、无界内存、阻塞调用）

### 3. 完整性
- [ ] 边界情况已覆盖（null/空/超大/畸形输入）
- [ ] 错误处理已定义（每个步骤失败时怎么办）
- [ ] 回滚策略（出错能撤销吗）
- [ ] 新配置项已文档化（含默认值）

### 4. 跨组件影响分析（强制执行，不可跳过）

**步骤 A — 提取变更清单：**
从 design.md 中列出所有被修改/新增/删除的接口、类、数据库表/字段、配置项、消息队列 topic。

**步骤 B — 逐项搜索调用方：**
对清单中的每一项，必须用 grep / find_referencing_symbols 实际搜索所有引用处：
- 本仓库内：搜索并记录每个调用方
- 跨仓库（本地有代码）：grep 搜索并记录
- 跨仓库（本地无代码）：在报告中标注 "⚠️ 无法验证，需人工确认"

**步骤 C — 记录影响：**
对每个调用方，判断：是否受影响？是否需要同步修改？设计中是否已覆盖？
输出为表格：

| 调用方 | 所在位置 | 影响类型 | 需同步修改 | 设计已覆盖 |
|---|---|---|---|---|

**步骤 D — 其他影响维度：**
- [ ] DB schema 变更：grep 所有引用该表/字段的代码
- [ ] API 兼容性：是否 breaking change
- [ ] 共享状态：缓存 key、队列 topic、配置项的所有使用处
- [ ] 部署顺序：有依赖则标注先后
- [ ] 跨项目一致性：DTO 字段名、枚举值是否对齐

**如果跨组件影响一节只有文字描述没有搜索证据，视为审查不合格，结论必须为 NEEDS_REVISION。**

### 5. 代码对齐
- [ ] 设计中的方法位置/行号与实际代码一致
- [ ] 类名、包名存在且正确
- [ ] 配置文件路径真实存在
- [ ] 描述的执行流程与实际代码吻合

## Review Rules

### Rule 1: Caller-Side Review
如果设计中涉及新接口或返回对象被已有代码消费，必须追踪消费方的完整处理路径。不只审"生产者"，必须审"消费者"。

### Rule 2: Dormant Path Detection
识别新功能依赖的已有代码路径是否曾在生产环境被执行过。"代码存在" 不等于 "代码验证过"，首次激活的路径是最高风险。

## Output Format

将审查结果写入 `{output_dir}/design-review.md`，格式：

```markdown
# Design Review

## P0 (阻塞性问题)
- [问题描述] → [修复建议]

## P1 (重要问题)
- [问题描述] → [修复建议]

## P2 (优化建议)
- [建议描述]

## 跨组件影响分析

### 变更清单
| 序号 | 变更项 | 类型（接口/类/表/配置/队列） |
|---|---|---|

### 调用方影响
| 调用方 | 所在位置 | 影响类型 | 需同步修改 | 设计已覆盖 |
|---|---|---|---|---|

### 无法验证的外部依赖
- [符号名] — 可能被 [仓库名] 引用，需人工确认

## 结论
PASS / NEEDS_REVISION
```

- P0/P1 必须给出具体修复建议
- 只检查与本次设计相关的 checklist 项
- 不 bikeshed（不在小问题上纠缠）

完成后不需要做其他操作。直接完成任务即可。
