# Plan Review Checklist

## A. 覆盖度（设计 -> 计划追溯）
- [ ] 设计文档中的**每个功能点**都有对应任务？
- [ ] 逐一核对设计文档章节，标记已覆盖/未覆盖？
- [ ] 错误处理、边界情况在设计中提到的，都有对应任务？
- [ ] 非功能需求（性能、安全、监控）有对应任务？

## B. 任务粒度（INVEST 原则）
- [ ] **Independent**: 每个任务可独立执行和验证？
- [ ] **Negotiable**: 描述"做什么"而非"怎么做"？
- [ ] **Valuable**: 每个任务完成后产出可验证的价值？
- [ ] **Estimable**: 任务描述足够详细，可以估算工作量？
- [ ] **Small**: 每个任务 2-10 分钟可完成？
- [ ] **Testable**: 每个任务有明确的完成标准？

## C. 依赖关系
- [ ] 所有任务间依赖显式标注（blocks / blocked-by）？
- [ ] 关键路径（最长依赖链）已识别？
- [ ] 可并行的任务组已标注？
- [ ] 无循环依赖？

## D. 任务描述质量
- [ ] 每个任务包含：要修改的文件路径？
- [ ] 每个任务包含：具体改动描述？
- [ ] 每个任务包含：完成标准？
- [ ] 没有模糊任务（如"优化性能"、"改善代码质量"）？

## E. 完整性
- [ ] 包含单元测试任务？
- [ ] 包含集成测试任务（如需要）？
- [ ] 包含配置变更任务（如需要）？
- [ ] 任务顺序合理（基础设施 -> 共享组件 -> 功能实现 -> 测试）？
- [ ] 不确定的部分有 Spike 任务？

## F. 拆分质量（SPIDR）
- [ ] **Spike**: 不确定的部分单独拆为调研任务？
- [ ] **Path**: 不同用户路径分别拆任务？
- [ ] **Interface**: 复杂 UI 分层拆任务？
- [ ] **Data**: 不同数据类型/格式分别处理？
- [ ] **Rules**: 复杂业务规则拆分（先核心逻辑，再约束校验）？
