# E2E Test Review Checklist

## A. 测试设计质量
- [ ] 使用了等价类划分（有效类 + 无效类）？
- [ ] 使用了边界值分析（边界值、次边界值）？
- [ ] 复杂条件组合用决策表覆盖？
- [ ] 有状态机场景时用状态转换覆盖？

## B. 集成入口测试（强制项）
- [ ] 至少一个测试从调用方入口发起（Handler/Controller/CLI），而非直接调用新组件？
- [ ] 从入口到输出的完整数据流被验证？
- [ ] 新代码的输出在已有管线中被正确传递和处理？

## C. 覆盖完整性
- [ ] design.md 中的每个功能点至少有一个正向测试？
- [ ] 每个功能点至少有一个负向测试（或标注了跳过原因）？
- [ ] code-review.md 中标记为"未验证"的 dormant path 有对应测试？
- [ ] 覆盖矩阵已填写，无遗漏功能点？
- [ ] 测试用例数 >= 功能点数 x 2（或有充分解释）？

## D. 测试用例质量
- [ ] 每个测试用例可独立执行（无隐含依赖顺序）？
- [ ] 前置条件明确具体（不写"系统正常运行"）？
- [ ] 测试步骤明确具体（不写"输入有效数据"）？
- [ ] 预期结果可客观验证（不写"系统正常工作"）？
- [ ] 验证方式明确（断言 / 数据库查询 / 日志检查 / 文件检查）？

## E. 测试实现质量
- [ ] 测试文件存在且可编译？
- [ ] 测试实际被执行（有 PASS/FAIL 结果，非"待执行"）？
- [ ] 断言有意义（不是 `assertTrue(true)` 或空断言）？
- [ ] 测试数据隔离（不依赖外部状态、不污染共享数据）？
- [ ] 测试执行结果记录在 e2e-test-results.md 中？

## F. 反作弊验证
> auto-dev 特有：防止 AI 编造"通过的测试"
- [ ] git diff 中存在新增的测试文件（*.test.ts / *Test.java 等）？
- [ ] e2e-test-results.md 中有实际执行关键词（PASS/FAIL/passed/failed），而非只有"待执行"？
- [ ] 测试失败时有具体的错误信息（不是笼统的"测试未通过"）？
