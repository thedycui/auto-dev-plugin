# Acceptance Review Checklist

## A. 验收标准提取
- [ ] AC 来源为 design.md 中的验收标准章节？
- [ ] 若 design.md 无显式 AC，已从设计目标和改动清单中提取？
- [ ] 每条 AC 的描述清晰、可客观验证？
- [ ] 没有自行编造超出 design.md 范围的 AC？

## B. 验证方法选择（按优先级）
- [ ] 优先使用代码验证（读源码确认逻辑实现）？
- [ ] 有对应测试的 AC 使用测试验证（指出具体测试用例名）？
- [ ] 可运行的场景使用运行验证（构造输入、检查输出）？
- [ ] 每条 AC 的验证方式明确记录？

## C. 验证完整性
- [ ] design.md 中的每条 AC 都有对应验证结果（PASS/FAIL/SKIP）？
- [ ] 无遗漏的 AC（对照 design.md 逐条核查）？
- [ ] PASS 有证据（代码位置 / 测试用例名 / 运行输出）？
- [ ] FAIL 有具体缺失点和修复建议？
- [ ] SKIP 有原因说明（需要集成环境 / 需要外部服务 / 无法自动化验证）？

## D. 报告质量
- [ ] 报告包含 AC 表格（AC编号 / 描述 / 验证方式 / 结果 / 证据）？
- [ ] 报告包含通过率统计（X/Y PASS, Z FAIL, W SKIP）？
- [ ] 报告包含明确结论（PASS / FAIL）？
- [ ] acceptance-report.md 文件已写入指定目录？

## E. 边界约束
- [ ] 不做 AC 之外的额外验证（不 scope creep）？
- [ ] 不评价代码风格（聚焦功能是否满足）？
- [ ] 不重新审查已在 Phase 4 审过的代码质量问题？
- [ ] FAIL 的结论仅基于 AC 未满足，不基于主观判断？
