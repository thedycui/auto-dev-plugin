# Design Review Checklist

## A. 功能完备性
- [ ] 所有功能需求都有对应的设计组件？
- [ ] 非目标（Non-Goals）明确定义？
- [ ] 需求到设计组件的追溯关系完整？
- [ ] 验收标准（AC-N）已定义，且每条可客观验证？
- [ ] 验收标准覆盖所有核心功能点（含正向和负向场景）？

## B. 方案选型
- [ ] 至少评估了 2 个方案？
- [ ] 每个方案的优劣有量化对比（不只是定性描述）？
- [ ] 选择理由清晰，读者能理解"为什么选A不选B"？
- [ ] 是否存在 Golden Hammer（一种技术打天下）？
- [ ] 是否过度设计（过多抽象层、过多设计模式）？

## C. 数据模型
- [ ] 数据结构满足当前需求？
- [ ] 数据量级评估（百/万/亿级别）？
- [ ] 数据生命周期（创建、更新、归档、删除）？
- [ ] 索引策略（查询模式 -> 索引设计）？

## D. 可靠性 & 容错
- [ ] 故障模式分析（每个外部依赖挂了会怎样）？
- [ ] 重试/补偿/幂等策略？
- [ ] 数据一致性保证（强一致 / 最终一致 / 哪些场景各用哪种）？
- [ ] 超时设置（每个外部调用的超时值）？
- [ ] 降级方案（核心功能不可用时的兜底）？

## E. 安全性
- [ ] 输入验证策略（白名单优先）？
- [ ] 认证 & 授权方案？
- [ ] 敏感数据处理（传输加密、存储加密、日志脱敏）？
- [ ] SQL 注入 / XSS / CSRF 防护？

## F. 性能
- [ ] 性能目标（响应时间、吞吐量、并发数）？
- [ ] 热路径识别和优化策略？
- [ ] 缓存策略（缓存什么、过期策略、一致性）？
- [ ] 资源使用预估（CPU、内存、磁盘、网络）？

## G. 可维护性 & 可操作性
- [ ] 系统由松耦合的模块组成？
- [ ] 变更影响可评估？
- [ ] 可测试性（能写单元测试、能 mock 外部依赖）？
- [ ] 监控方案（关键指标、告警阈值）？
- [ ] 日志方案（关键操作留痕，不记敏感信息）？
- [ ] 部署方案（灰度？蓝绿？滚动？）？
- [ ] 回滚方案？

## H. 文档质量
- [ ] 有上下文说明（为什么做这个）？
- [ ] 有架构图 / 流程图？
- [ ] 技术约束和假设显式声明？
- [ ] 迁移路径（如果替换旧系统）？

## I. 结构化 AC 审查
- [ ] acceptance-criteria.json 文件已生成且 schema 合法？
- [ ] 每条 AC 都有 layer 标注（structural / test-bound / manual）？
- [ ] structural 类型的 AC 断言是否合理（path 是否可能存在、pattern 是否正确）？
- [ ] manual 占比是否 <= 40%？
- [ ] test-bound 类型的 AC 描述是否足够具体，让 Phase 5 能写出对应测试？
